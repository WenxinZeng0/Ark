# 方舟号 · 永恒执勤 —— 任务/物品/选择 逻辑说明

这份文档梳理代码里四个任务的触发顺序、每个任务里"必须先查看哪个物品才会解锁哪个选项"的依赖关系，以及每个选择会导致什么后果（死亡 / 受伤 / 正常推进）。代码位置用 `行号` 标注，方便对照原文件 `ark-v2 2.html`。

---

## 整体流程

```
休眠舱(sleep)
  → 主走廊(corridor)  ← 所有任务的枢纽，警报会指向下一个该去的房间
      → 任务1：B3区 (b3)        10:00 液氮泄漏
      → 任务2：中央机房 (machine) 14:00 火灾
      → 任务3：引擎廊→引擎舱 (engcorr → engine) 17:00 引擎故障
      → 任务4：休眠廊→休眠舱深处 (cryocorr → cryo) 20:00 气阀卡死 → 结局
```

每完成一个任务，会 `unlockRoom()` 解锁下一个任务房间，并把 `G.taskPhase` 切到下一个阶段。**任务必须按 1→2→3→4 顺序做**，没有解锁的房间在地图上是灰色锁住状态，点不动。

`G.t1_done / t2_done / t3_done / t4_done` 记录任务完成状态；`G.hp`（生物体完整度）降到 0 会触发死亡画面 `triggerDeath()`，并重开新的一轮（"周期 N+1"）。

## 任务1：B3区 · 液氮泄漏（10:00）

代码：`rooms.b3()` 1258行，`chooseGate()` 1877行，`chooseValve()` 1911行

**阶段细分**（`G.taskPhase`）：`task1_alert`（走廊有警报）→ 进入B3后等同 `task1_gate`（闸门关）→ 选择后进入 `task1_valve`（阀门关）→ `finishTask1()` 完成。

### 第一关：闸门
入口被闸门卡住，缝隙20厘米。场景里可查看的物品：
- `gate` 应急隔离闸（说明缝隙边缘有冰晶传感器）
- `gate_panel` 手动解锁面板（说明解冻后可用）
- `heater_gun` 备用加热枪（说明电量只够用一次）

这三个物品**只是文字说明，不会解锁/隐藏任何选项**——闸门关的4个选择按钮始终都可见：

| 选择 | 后果 |
|---|---|
| 侧身挤过闸门缝隙 | **死亡**（碰到冰晶传感器，闸门锁死） |
| 用加热枪解冻手动面板 | 正常进入下一关（阀门），无伤 |
| 用加热枪直接喷闸门 | 加热枪浪费、扣HP 10%，但有补救剧情仍可进入下一关 |
| 返回走廊找撬棍 | 仅是 `travelTo('corridor')`，不会有撬棍剧情，相当于原地打转 |

### 第二关：主阀
场景物品：
- `valve` 主阀旋钮
- `gloves` 隔热手套
- `emergency_valve` 紧急截断阀（**位置较高，需要垫脚**）
- `b3_toolbox` 工具箱（**可当踏脚台**）

**关键依赖**：「用工具箱垫脚，操作紧急截断阀」这个选项按钮一开始带 `unavailable`（半透明、点不动）样式，**必须先查看过 `emergency_valve` 或 `b3_toolbox` 任意一个**，才会被代码移除 unavailable 状态变得可点（1308行 `if (!G.examined.has('emergency_valve') && !G.examined.has('b3_toolbox'))`）。

| 选择 | 后果 |
|---|---|
| 赤手操作旋钮 | 扣HP 30%（严重冻伤），完成任务 |
| 戴手套操作旋钮 | 扣HP 15%，完成任务 |
| 垫脚用紧急截断阀（需先查看上述物品解锁） | **无伤**完成任务，最优解 |

完成后解锁机房 `unlockRoom('machine')`，14:00 触发任务2。

## 任务2：中央机房 · 火灾（14:00）

代码：`rooms.machine()` 1316行，`chooseFire()` 1962行，`chooseRepair()` 2014行

### 第一关：灭火
场景物品：
- `extinguisher` 灭火器（说明只够撑90秒×2，"火已蔓延三处"，暗示不够用）
- `vent_machine` 通风管道入口（说明可以进去关进气阀灭火）
- `console` 主控台（**藏着隐藏代码输入框**，只有标注没给代码）
- `notice_recall`（走廊告示板的记忆，**前提是任务1之前去过走廊查看过 `notice` 告示板**，否则只显示一句模糊提示）

**关键依赖**："输入隔离代码0425"按钮（`#fire-code-btn`）默认 `unavailable`，**只有上一轮死亡留下的遗书（`prevLetter`）里提到过"0425"这个代码才会解锁**（1347行 `prevLetter.some(l => l.includes('0425'))`）。也就是说**第一次游玩永远点不开这个选项**，要等某一轮因为用过代码0425死亡/记录后，下一轮的遗书才会提到它。
"触发全舰紧急降压"按钮永远是 unavailable（需要双人授权，设计上就是不可选项）。

| 选择 | 后果 |
|---|---|
| 拿灭火器冲入扑救 | **死亡**（灭火器不够用，被困） |
| 进入通风管道关闭进气阀 | 扣HP 20%（吸入浓烟），正常进入"灾后处置" |
| 输入代码0425（需解锁） | 不扣HP，秒灭火，但 `G.civ`（公民存活率）直接掉到60%，并触发"B区氧气切断，1万余人死亡"的剧情提示——文本暗示这是"系统Bug"，但结局画面会戳破这个谎言 |

### 第二关：灾后处置（三选二，必须放弃一个）
场景物品：`repair_X`（空气过滤膜）、`repair_Y`（数据备份模块）、`repair_Z`（B区阀门校准，系统显示"正常"）

这里没有解锁限制，三个选项都能直接点，纯剧情后果：

| 放弃项 | 后果 |
|---|---|
| 放弃X 空气过滤膜 | HP上限永久-10%（代码里简化处理成一条日志，没有真正生效数值） |
| 放弃Y 数据备份模块 | 本轮死亡后的遗书会被"腐蚀"——`generateLetter()` 里专门为这个分支在遗书中间插入一行"数据损坏"，并且遗书会缺失部分线索句 |
| 放弃Z B区阀门校准 | 表面"状态正常"，实为埋下后续秘密日志（伏笔，无即时后果） |

完成后解锁引擎廊 `unlockRoom('engcorr')`，17:00 触发任务3。

## 任务3：引擎廊 → 引擎舱 · 引擎故障（17:00）

代码：`rooms.engcorr()` 1369行，`rooms.engine()` 1407行，`chooseEngPrep()` 2051行，`chooseEngOp()` 2070行

### 第一关：进入前的准备（引擎廊）
场景物品：
- `rad_pills` 防辐射药片（**已过期49个月**，说明书直接写"可能损伤倍增"）
- `heat_blanket` 绝热毯（说明"会显著降低行动速度"）
- `eng_console` 主控台（**有个没标注的第三断路器**，划痕暗示有人动过）

**关键依赖**："拉下第三个断路器"按钮（`#eng-breaker-btn`）默认 `unavailable`，解锁条件是**满足下面任一个**（1399行）：
- 上一轮遗书提到过"断路器"这个词，或
- 本轮查看过 `eng_console` 这个物品

| 准备选择 | 后果 |
|---|---|
| 服用防辐射药片 | 药片过期，**会在进入引擎舱操作时直接死亡**（药物+辐射叠加） |
| 裹绝热毯 | 拖慢速度，**进入引擎舱操作时直接死亡**（辐射超量） |
| 拉下断路器（需解锁） | 辐射窗口从20分钟压缩到8分钟，是去重启时**唯一不会自动死亡**的准备方式，但HP消耗仍取决于第二关怎么操作 |
| 什么都不做直接冲 | 标准20分钟窗口，HP消耗最高 |

注意：选"药片"或"绝热毯"时，进入引擎舱选哪个重启方式都救不了，是必死分支。

### 第二关：引擎舱重启
场景物品：`eng_toolbox` 工具箱（**里面有简化操作流程，能少待6分钟**）

**关键依赖**："使用工具箱简化流程"按钮（`#eng-simplified-btn`）默认 `unavailable`，**必须先查看过 `eng_toolbox`** 才解锁（1431行）。

| 重启选择 | 后果 |
|---|---|
| 标准程序操作 | 耗时长，HP扣得更多 |
| 简化流程操作（需先查看工具箱解锁） | 耗时短，HP扣得更少 |
| 跳过点火确认直接撤离 | **死亡**（轨道修正失败，撞上陨石带） |

实际扣血量由 `t3_prep` × `t3_engine_method` 组合决定（2084-2121行）：

| 准备方式 \ 操作方式 | 标准程序 | 简化流程 |
|---|---|---|
| 断路器（8分钟窗口） | -35% HP | -25% HP |
| 无防护（20分钟窗口） | -70% HP | -55% HP |
| 药片 / 绝热毯 | 直接死亡，与操作方式无关 | 同上 |

另外有一条隐藏致死判定：如果 `t3_prep==='raw'` 且选"标准程序"，扣血后HP正好≤5%也会触发死亡（"耗尽了一切"）。

完成后解锁休眠廊 `unlockRoom('cryocorr')`，20:00 触发任务4。

## 任务4：休眠廊 · 气阀卡死（20:00）→ 结局

代码：`rooms.cryocorr()` 1438行，`rooms.cryo()` → `triggerEnding()` 2159行

场景物品（**纯剧情/伏笔，不影响任何选项解锁**）：
- `elec_box` 电网分电箱（耗电量几乎是0，与"维持30万人冷冻"的说法矛盾）
- `air_pipe` 循环回气管（管道是干的，没有该有的冷凝水/气体波动——暗示"冷冻舱"根本没人活着）
- `plaques` 铭牌列表（墙上一排和"你"同名的铭牌，对应之前所有轮回角色）

这里只有两个按钮：返回走廊 / 强行撬开舱门。撬门直接 `triggerEnding()`，没有选择分支——这是固定结局，揭示休眠区里堆满的是历代"执勤员"的尸体，而不是冷冻的平民。如果任务2用过代码0425，结局文案会额外提到"最近一具尸体颈部有缺氧痕迹"，呼应0425造成的死亡。

## 跨周期机制（死亡后怎么继续）

1. 死亡 `triggerDeath()` 会调用 `generateLetter()`，把本轮做过的关键选择编译成几行"遗书"文字，存进 `prevLetter`。
2. 下一轮开局时，遗书内容会通过 `prevLetter.some(l => l.includes(...))` 这种文本匹配，**反向解锁任务2的代码0425选项、任务3的断路器选项**。也就是说有些选项第一轮根本拿不到，需要先死一次，让遗书"剧透"给下一任。
3. `chooseRepair('Y')` 放弃数据备份模块会让遗书"腐蚀"（中间插入一行读不出来的内容），可能让该轮遗书丢失关键提示。

## 当前已知的两个解锁bug（已修复，仅供参考）

- B3区第一次进入时 `taskPhase` 一直停在 `task1_alert`，原本判断只认 `task1_gate`，导致点了没反应——已让 `task1_alert` 和 `task1_gate` 共用同一渲染分支。
- 引擎舱房间 `engine` 从来没有被 `unlockRoom()`，导致准备阶段4个选项点完都卡住——已在 `startTask3()` 里补上解锁。
